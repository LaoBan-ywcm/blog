移动Web手册 (视口)

发表于 2015-09-12
像素

一个像素就是计算机屏幕能显示一种特定颜色的最小区域。

  • 设备像素：设备屏幕的物理像素，任何设备的物理像素的数量都是固定的。


  • CSS像素：为Web开发者创造的，在CSS(和JavaScript)中使用的一个抽象的层。

用户缩放的越大，一个CSS像素覆盖的设备像素就越多。
当使用CSS和JavaScript的时候，并不在意一个CSS像素跨越了多少个设备像素。实际上几乎用不上设备像素，唯一的例外就是screen.width/height。

视口：在CSS标准文档中，被称为初始包含块。

布局视口：

在手机上，该视口与移动浏览器屏幕宽度不再相关联，而是完全独立，CSS布局会根据它来计算，并被它约束。可看作初始包含块的宽度。
document.documentELement.clientWidth/Height

视觉视口：

用户屏幕中看到区域的CSS像素。视觉视口和设备屏幕一样宽。用户可以通过缩放来操作视觉视口（它的CSS像素数量随用户缩放改变），同时不会影响布局视口，布局视口宽度不变。
window.innerWidth/Height。

理想视口：

是针对设备来说最理想的布局视口尺寸。只有当网站是为手机准备的时候才应该使用理想视口。

screen.width/height返回理想视口的尺寸，但有很严重的浏览器兼容性问题。

定义理想视口是浏览器的工作。因此，同一设备上的不同浏览器拥有不同的理想视口。

理想视口也会随设备转向发生改变。有一个重要的例外：ios 中的 Safari。这个浏览器不会根据设备旋转调整理想视口：它顽固地保持在320px（iphone）或 768px（iPad）。

缩放

缩放会影响手机的视觉视口的尺寸。放大使视觉视口变得更小，因为屏幕上显示的CSS像素更少了。而缩小会让视觉视口更大，因为屏幕上的CSS像素更多了。因此缩放程度和视觉视口的大小是逆相关的：放的越大，视觉视口就越小。

但是视口的设备像素是不变的。

浏览器根据理想视口的大小来计算缩放程度。

移动端缩放，布局视口不变。缩放不会导致css布局被重新计算。在手机上会经常发生缩放，并且手机的处理器工作很慢，电池消耗很快，因此不重新进行布局对性能来说有很大的好处。

禁止用户缩放：<meta name="viewport" content="user-scalable=no">，但是这种行为是不可宽恕的邪恶。

一方面，它是设备每英寸内点的个数； 另一方面，CSS和JavaScript中也有分辨率的概念，但实际上它们并不是一回事。

meta 视口

meta 视口标签存在的主要目的是让布局视口的尺寸和理想视口的尺寸匹配。

共有5个名/值对：

  1. width：设置布局视口的宽度为特定的值，例：width=device-width。
  2. init-scale：设置页面的初始缩放程度和布局视口的宽度
  3. minimum-scale：设置了最小缩放程度（用户可缩小的程度）
  4. maximum-scale：设置了最大缩放程度（用户可放大的程度）
  5. user-scalable：是否阻止用户进行缩放。请无视它。

一般，initial-scale=1与width=device-width的效果是一样的。
但一个意外发生在Safari上，只设置device-width浏览器不会根据设备旋转调整理想视口，它会保持一个固定值；当你设置initial-scale=1时，理想视口的尺寸会随着屏幕的旋转改变。

媒体查询

媒体查询其实就是CSS的if语句。

  • 媒体查询都需要一个媒体类型，通常使用all是最好的。
  • 一般在媒体查询中总是使用min-或max-前缀，因为通常情况下，不应关心一个准确的值，而是关心一个范围。
  • 虽然没有定义正式的单位，但媒体查询的单位是像素。

and是逻辑运算符and，逗号 是逻辑运算符or。

在移动端，根元素的字体大小在你缩放的时候是不会变的。1em的宽度还是同样多的css像素。所以，在移动端，em并不比像素优秀。

device-width和device-height媒体查询使用了JavaScript提供的screen.width/heigth，不用管浏览器使用的是哪个定义，问题很多。都要避免使用。

CSS

移动端上：

  • position: fixed有很多问题，慎用；
  • 忘掉overflow: auto、overflow-scrolling、background-attachment；
  • 不能使用vw和vh；
  • :hover、:active无法真正在移动端工作；:focus可以很好的工作；
  • 建议不要在移动设备上使用过多的动画效果；
